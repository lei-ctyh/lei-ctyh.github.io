Markdown 样式测试文档

用途：将本文导入或复制到飞书文档，用来对比 Obsidian、Minimal 主题、飞书云文档之间的排版差异

重点观察：标题层级、段落行高、列表缩进、代码块、引用块、表格、图片位置、图片尺寸、图文混排。

1. 基础正文

这是一段普通正文，用来观察默认字号、行高、段落间距和正文宽度。理想的阅读效果应该是文字不拥挤、行距舒适、每行长度适中，长时间阅读时不会产生明显疲劳。

这是第二段正文。这里包含一些中文、English words、数字 1234567890，以及常见标点符号：逗号，句号。冒号：分号；括号（内容）和引号“示例”。

这是一段较长的正文，用来测试自动换行效果。如果正文容器过宽，读者的视线移动距离会变长；如果正文容器过窄，换行会过于频繁。飞书文档的正文宽度通常比较克制，这也是它看起来舒服的重要原因之一。

2. 行内样式

这是一段包含 加粗文本、斜体文本、加粗斜体、删除线、inline code、普通链接 的正文。

中文技术文档里经常会出现变量名，例如 fileKey、fileUrl、folder、uploadPath。这些行内代码应该足够醒目，但不应该打断阅读节奏。

3. 标题层级

H1 一级标题：页面主标题

一级标题通常只应该在页面顶部出现一次，用来观察主标题字号、字重和上下间距。

H2 二级标题：主要章节

二级标题适合划分主要章节，应该有明显的视觉权重。

H3 三级标题：小节标题

三级标题适合进一步拆分内容，应该比正文突出，但不要过于夸张。

H4 四级标题

四级标题常用于技术文档中的细节说明。

H5 五级标题

五级标题测试较低层级标题的可读性。

H6 六级标题

六级标题测试最小标题样式。

11.5 左右图文混排：左图右文

这是一段用于测试左图右文的正文。图片如果浮动成功，文字会从图片右侧开始环绕；如果不支持浮动，图片会独占一行。飞书导入 Markdown 时不一定保留 align="left"，因此这里主要用于观察兼容性。

这是一段继续补充的正文，用来测试多行文字环绕时的行高和右侧留白。若效果不稳定，建议在飞书里使用表格或分栏来实现稳定的图文排版。


11.6 左右图文混排：右图左文

这是一段用于测试右图左文的正文。图片如果浮动成功，文字会在图片左侧排列；如果导入后浮动失效，说明目标平台对 HTML 图片布局支持有限。

在技术文档里，右图左文适合放界面截图说明，但如果兼容性不好，推荐改用表格方式实现。

16. 综合样例：文件上传模块说明

16.1 总体原则

本项目文件上传的核心原则：

  1. 业务表只保存 fileKey。
  2. 页面展示使用 fileUrl。
  3. fileUrl 由后端统一生成。
  4. 前端不要手工拼接 /upload/${fileKey}。
  5. 上传目录由 folder 参数决定。
  6. 本地存储和云存储共用同一套 fileKey 规则。

推荐将所有文件访问入口收口到后端服务，避免前端直接依赖具体存储实现。

17. 视觉检查清单

  • 正文字号是否舒适。
  • 正文行高是否接近飞书的 1.8。
  • 正文宽度是否接近 820px。
  • 标题上下间距是否清晰。
  • 列表缩进是否自然。
  • 代码块背景是否足够轻。
  • 表格是否清晰但不沉重。
  • 图片默认位置是否符合预期。
  • 左右图文混排是否稳定。
  • 复制到飞书后是否有明显样式丢失。
